网关将下列路径原样转发到后端（内网 HTTP）。响应统一为 `{"code":0,"message":"","data":{}}`。

| 方法 | 路径                                 | 请求体                                                | 说明                                            |
| :--- | :----------------------------------- | :---------------------------------------------------- | :---------------------------------------------- |
| POST | `/api/auth/login`                    | `{"username":"","password":""}`                       | 校验登录并写入 Redis 会话                       |
| POST | `/api/auth/logout`                   | —                                                     | 删除 Redis 会话并清除 Cookie                    |
| GET  | `/api/payment/orders/{order_sn}`     | —                                                     | 订单确认页（需登录态）                          |
| POST | `/api/payment/confirm`               | `{"order_sn":"","agreement_checked":true}`            | 继续支付，返回 `payment_session_id`、`qr_token` |
| POST | `/api/users/real-auth/complete`      | `{"email":"","id_card":"","return_url":""}`           | 补全实名（需登录态）                            |
| GET  | `/api/payment/page?token={qr_token}` | —                                                     | 扫码支付页                                      |
| POST | `/api/payment/attempt/init`          | `{"qr_token":""}`                                     | 申请 `request_id`                               |
| POST | `/api/payment/pay`                   | `{"request_id":"","qr_token":"","login_password":""}` | 提交支付                                        |

## 当收到了网关发来的请求后，对各个请求的处理流程：

**`POST /api/auth/login` / `POST /api/auth/logout`**

1. 登录时校验账号状态与密码，生成高强度随机 `session_id`。
2. 使用后端 Redis 读写账号执行 `SET payment:session:{session_id} {user_id} EX {ttl}`。
3. 登出时读取网关转发的 Cookie，使用后端账号删除对应会话键。
4. `Set-Cookie` 经网关原样透传给浏览器。

**`GET /api/payment/orders/{order_sn}`**

1. 取 `X-User-Id` → 查 `orders`，校验存在且 `buyer_id` 匹配。
2. 查 `accounts` → 返回订单与账号展示信息。

**`POST /api/payment/confirm`**

1. 取 `X-User-Id`，校验 `agreement_checked == true`。
2. 查 `users`，校验状态正常、已实名；未实名 → `40910`。
3. 查 `orders` / `accounts`，校验归属与状态。
4. 创建或复用 `payment_sessions`，生成 `qr_token`。
5. 返回 `payment_session_id`、`qr_payload`、`websocket_url`。

**`POST /api/users/real-auth/complete`**

1. 取 `X-User-Id`，校验邮箱与身份证格式。
2. 标准化 → 算 hash、脱敏 → 更新 `users`。
3. 返回 `redirect_url`。

**`GET /api/payment/page?token={qr_token}`**

1. `verifyPaymentQrToken`：用服务端 `JWT_SECRET` 重算 HMAC 签名并比对 → 失败则非后端签发或已被篡改；通过后校验 `exp`，取出 `payment_session_id`。
2. 查 `payment_sessions`：存在、`status=0`、`expire_at > NOW()`。
3. 查 `orders` / `accounts`，校验未支付、账号可交易 → 返回支付页数据。

**`POST /api/payment/attempt/init`**

1. 验签 `qr_token` → 查 `payment_sessions`，校验会话与订单/账号状态。
2. 复用或生成 `request_id`，插入 `payment_attempts`（`result=0`）。
3. 返回 `request_id`。

**`POST /api/payment/pay`**

1. 查 `payment_attempts` 做幂等（已成功直接返回，已失败返回原原因）。
2. 验签 `qr_token`，校验会话/订单/账号/用户状态。
3. 校验 `login_password` 与 `users.password_hash`。
4. 事务更新 `orders` / `accounts` / `payment_sessions` / `payment_attempts`。
5. 响应页面 B，通知网关推送 `PAYMENT_RESULT`。

## 整个业务流程：

1. **页面 A 下单前确认**：`GET /api/payment/orders/{order_sn}` → 展示订单与账号信息。
2. **页面 A 提交支付信息**：`POST /api/payment/confirm` → 校验用户、订单、账号、实名。
   - 未实名 → 浏览器获取3开头的下一跳进入页面C完成实名信息的提交 → `POST /api/users/real-auth/complete`
   - 通过 → 创建/复用 `payment_session_id`，生成 `qr_token` → 页面 A 连 WebSocket 等结果，展示二维码。
3. **页面 B 扫码支付**（手机扫码进入支付页，按序自动请求，前一步成功才进行下一步）：
   - 进入页面 → 自动 `GET /api/payment/page?token=`，后端校验 token/会话/订单/账号。
   - 成功 → 自动 `POST /api/payment/attempt/init`，获取 `request_id`。
   - 成功 → 页面展示登录密码输入框；用户填写后 `POST /api/payment/pay`。
   - 后端校验密码、更新 SQL 表 → 响应页面 B；同时通知网关，页面 A 收到支付结果。
4. **支付成功**：后端事务更新 `orders` / `accounts` / `payment_sessions` / `payment_attempts` → 通知网关推送 `PAYMENT_RESULT` → 页面 A 展示成功。

## 电脑端如何保持登录态：

适用页面 A、页面 C 等同域跳转；页面 B 扫码不走此机制，仍用 `qr_token`。

1. **登录**：网关转发请求 → 后端校验账号密码 → Redis 存 `session_id → user_id` 并设置 TTL → 返回 Cookie。
2. **跳转**：浏览器访问后续页面自动带 Cookie → 网关使用只读账号查 Redis → 内网转发时加 `X-User-Id`。
3. **后端业务接口**：不读 Cookie，只信网关传来的 `X-User-Id` 作为 `current_user_id`。
4. **登出**：网关转发 Cookie → 后端删除 Redis 会话并返回清 Cookie 响应头。

## qr_token存的是什么：

JWT 字符串（`header.payload.signature`），**不落库**。payload 仅含：

| 字段                 | 含义            |
| :------------------- | :-------------- |
| `payment_session_id` | 支付会话 ID     |
| `iat` / `exp`        | 签发 / 过期时间 |
| `nonce`              | 随机串          |

订单号、金额等不在 token 里，后端验签后凭 `payment_session_id` 查库。二维码内容是带 `?token=` 的跳转 URL，不是加密，改 payload 会导致验签失败。
