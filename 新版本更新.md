# vNext 更新计划 — 并发模型升级

> 目标：以**最小可用**方式解决当前单线程 HTTP 阻塞与 `detach` 线程失控问题，提升并发处理能力，消除支付回调与网关 HTTP 之间的死锁风险。

---

## 1. 现状与问题

| 位置                            | 当前实现                                                   | 问题                                              |
| ------------------------------- | ---------------------------------------------------------- | ------------------------------------------------- |
| `backend/main.cpp`              | 单线程 `accept` → 同步读请求 → 同步处理 → 写响应 → `close` | 一个慢请求阻塞后续所有连接；`listen` backlog 偏小 |
| `gateway/main.cpp`（HTTP）      | 同上                                                       | 与 Backend 相同瓶颈；曾因此与支付回调形成死锁     |
| `gateway/main.cpp`（WebSocket） | 每连接 `std::thread(...).detach()`                         | 高并发下线程数不可控                              |
| `backend/payment_impl.hpp`      | 支付成功后 `std::thread(...).detach()` 通知网关            | 高并发支付时线程爆炸；无背压与队列上限            |

---

## 2. 优化目标

1. **HTTP 服务**：Backend 与 Gateway HTTP 改为 **epoll 事件驱动 + 固定大小 worker 线程池**。
2. **异步任务**：支付成功通知网关统一走 **共享线程池**，不再 `detach` 裸线程。
3. **连接参数**：提高 `listen` backlog；HTTP 响应支持 **Keep-Alive**（同连接多请求，减少短连接开销）。

---

## 3. 改造范围

### 3.1 新增公共模块 `common/thread_pool.hpp`

- 固定 worker 数量（默认 = `std::thread::hardware_concurrency()`，可通过 `.env` 配置）
- 接口：`submit(F&& f)` 投递任务；析构时等待队列清空
- Backend 与 Gateway 各自持有独立实例

### 3.2 新增公共模块 `common/http_server.hpp`

抽取 Backend / Gateway 重复的 HTTP 服务骨架：

```
createListenSocket(port, backlog)
  ↓
epoll 监听 server_fd
  ↓
accept 新连接 → 注册 EPOLLIN
  ↓
可读事件 → 读完整 HTTP 请求 → thread_pool.submit(handleRequest)
  ↓
worker 处理完毕 → 写响应 →
  └─ Connection: close：关闭 fd，从 epoll 移除
```

**关键参数（写入 `.env.example`）：**

| 配置项                            | 建议默认值 | 说明                     |
| --------------------------------- | ---------- | ------------------------ |
| `HTTP_WORKER_THREADS`             | 10         | worker 线程池大小        |
| `HTTP_LISTEN_BACKLOG`             | 128        | `listen()` backlog       |
| `HTTP_MAX_CONCURRENT_CONNECTIONS` | 256        | epoll 同时管理的连接上限 |
| `HTTP_READ_TIMEOUT_SECONDS`       | 10         | 单连接读超时             |

### 3.3 Backend 改造

**文件：** `backend/main.cpp`

- 删除主循环中的同步 `accept → read → handle → write → close`
- 改为调用 `http_server` 启动；回调中调用现有 `payment_impl::handleRequest`
- `listen` backlog 从当前值改为读取 `HTTP_LISTEN_BACKLOG`

### 3.4 Gateway HTTP 改造

**文件：** `gateway/main.cpp` 中 `runHttpServer`

- 与 Backend 同样接入 `http_server`
- 保留 **Gateway 入口 IP 白名单** 用的 `request.client_ip` 注入：
  - 在 `accept` 时通过 `sockaddr_in` + `inet_ntop` 取得 **TCP 直连对端地址**（peer address）
  - 浏览器流量下通常是 **Nginx 的 IP**（如 `127.0.0.1`），内网回调下是 **Backend 的 IP**
  - 供 `handleRequest` / `handlePaymentResult` 校验来源，**不是**手机用户的公网 IP
- **审计用的真实客户端 IP** 走 HTTP 头 `X-Forwarded-For`（Nginx 设置）→ `toForwardRequest` 转发 → Backend 写入 `payment_attempts.client_ip`（**已实现**，并发改造时勿破坏）

### 3.5 支付异步通知改造(这是通知网关的)

**文件：** `backend/payment_impl.hpp`、`backend/common/gateway_notify.hpp`

- 删除：
  ```cpp
  std::thread([...]() { notifyPaymentResult(...); }).detach();
  ```
- 改为：
  ```cpp
  thread_pool.submit([notify_session_id, notify_order_sn]() {
      payment_gateway_notify::notifyPaymentResult(notify_session_id, notify_order_sn);
  });
  ```
- 线程池与 HTTP worker 可共用同一池，或单独 `NOTIFY_WORKER_THREADS`（最小方案：共用 HTTP worker 池即可）

### 3.6 Gateway WebSocket

- **暂不重构** epoll 模型（长连接特性与 HTTP 不同，改动面大）
- 仅增加连接数上限保护：超过 `WS_MAX_CONNECTIONS` 时拒绝新连接并打日志
- 后续版本再考虑 WS 专用 epoll 线程

## 4. 实施顺序

```
Step 1  common/thread_pool.hpp          ← 无外部依赖，可先写单测
Step 2  common/http_server.hpp          ← 依赖 thread_pool
Step 3  backend/main.cpp 接入            ← 验证 Backend 并发
Step 4  gateway/main.cpp HTTP 接入       ← 验证网关转发
Step 5  payment_impl.hpp 线程池替换      ← 消除 detach
Step 6  .env.example 补充配置项          ← 文档同步
Step 7  现有 ctest 全量通过 + 手动压测     ← 验收
```

---

## 5. 验收标准

- [ ] Backend / Gateway HTTP 在 50 并发短连接下不出现单请求阻塞全局 accept
- [ ] 支付成功 → 网关 WS 推送链路正常（现有 `test_payment_result` 等用例通过）
- [ ] 100 次连续支付通知不产生超过 `HTTP_WORKER_THREADS` 数量的活跃线程，这里指的是后端通过线程池的方式往网关进行HTTP支付成功请求的推送。
- [ ] `HTTP_MAX_CONCURRENT_CONNECTIONS` 超限时新连接被拒绝并记录 WARN 日志
- [ ] CI（`.github/workflows/ci.yml`）全部通过
